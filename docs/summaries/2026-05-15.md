# Session summary — 2026-05-15

Short, focused day. One PR. Groq model swap.

## Completed

- **Groq model swap: `openai/gpt-oss-120b` → `llama-3.3-70b-versatile`** (PR #30, merged). 8K TPM free-tier on the old model was hitting the cap on almost every generation, even after the per-input compression + auto-retry layers from PR #25. Llama 3.3 70B free tier is **12K TPM** (+50%) with equivalent JSON-mode reliability for our ATS rewrite use case.
- Budget constants raised in `lib/llm.ts` to actually use the new headroom:
  - `TPM_BUDGET`: 8000 → 12000
  - `MAX_OUTPUT`: 5500 → 6000 (more room for full resume + cover letter)
  - Per-input caps in `app/api/generate/route.ts`: JD 1500 → 2500, resume 2000 → 4500 (3–4 page resumes now pass uncut)
- Home-page client-side soft caps (`TOKEN_BUDGET`, `JD_TOKEN_SOFT_CAP`, `SOFT_CAP`) updated to mirror.

`SAFETY_MARGIN` (1200), the `chars/3` estimator, and the 413 auto-shrink-retry path all kept — even with 50% more headroom, the estimator gap could still bite for URL/code-heavy inputs.

## Important changes

- **The model swap surfaced a new prompt issue.** Llama 3.3 is biased toward concise output and interprets "REWRITE the resume to be ATS-friendly" as "summarise into a tight 1-page resume." The user's 8-job resume came back as 2 jobs with 1 bullet each. Required PRESERVE COMPLETENESS rule (shipped the next day in PR #32). The old `gpt-oss-120b` happened to be verbose enough that completeness was implicit.

## Open problems (at end of day)

- The output verbosity / completeness issue is critical for product quality. Has to be fixed before users with longer resumes can use the tool.

Fix the prompt to preserve all jobs / skills / bullets explicitly. Add an explicit length constraint ("output should be within 10% of original length"). Add a self-check at the end of the rule block.
